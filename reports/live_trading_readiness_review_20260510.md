# LAOWANG 实盘准备度审查

审查日期：2026-05-10  
审查口径：当前目标不是自动下单，而是盘后生成信号，隔日人工执行；Relay 另按“盘后候选 + 集合竞价过滤 + 盘中触发 + 人工下单”处理。  
结论：项目已经有研究和半自动工作流骨架，但还不是可直接投入真钱的交易系统。

## 1. 当前状态

仓库已有：

- 数据入库：`getDataBaoStock.py` 支持日线和分钟线表结构，含增量水位、批量 upsert、分片拉取。
- 日常流程：`everyday.py` 负责日线增量和四个普通评分模型，`everydayReview.py` 负责 Relay 增量。
- 展示和提醒：`ui.py` 是只读 Web UI，并有每日自动任务；`tgBot.py` 支持 Telegram 推送。
- 回测和整改报告：`backtest_model_follow.py`、`optimize_relay_follow.py` 以及 2026-05-05 的可行性审查已经把“评分模型不是交易系统”这个核心问题暴露出来。
- 策略审计表：`strategy_signal_log`、`strategy_trade_journal` 已在 `init.py` 建表，但尚未接入每日流程。

当前最大事实：

- 普通模型严格 T+1 手动计划在 2025-01-01 至 2026-04-30 区间产生 23 条信号，但 0 笔通过买入过滤。
- Relay 传统日线回测有收益，但最大回撤过高；Relay hybrid 目前只是日线代理，不是真正分时回测。
- `stock_minute` 在最近回测环境里不存在，因此“集合竞价/分时触发”还没有真实数据闭环。

## 2. 距离实盘的判断

### 可以做的

现在可以作为盘后研究系统使用：

- 每天拉数据、评分、看 UI、收 Telegram 推送。
- LAOWANG / STWG 做观察池。
- Relay 做次日盯盘候选池。
- 普通模型继续生成次日计划，但严格过滤后没有买入就不买。

### 不能做的

现在不应该直接实盘买入：

- 不能把 `model_*_pool` 的排名当成买入指令。
- 不能把 Relay 的日线概率当成开盘买入指令。
- 不能自动下单。
- 不能放大仓位验证。

### 实盘距离

按“隔日人工下单、小资金、保守仓位”定义，距离约 2 到 4 周，前提是只做 Relay 小资金验证，并接受它先是纸面盯盘系统，不是自动交易。

按“稳定可复制、可持续加仓”的定义，距离至少 2 到 3 个月，因为必须补分钟/竞价特征、walk-forward、审计闭环、真实人工执行样本和风控熔断。

按“自动交易系统”的定义，距离很远，至少 3 到 6 个月以上；当前仓库没有券商交易网关、订单状态机、成交回报、撤单、重试、限价保护、账户级风控。

## 3. 主要缺口

### P0: 数据和执行口径

- 真实分钟线/竞价数据没有进入 Relay 模型。
- 盘中触发现在用日线 OHLC 代理，无法证明真实可成交。
- 停牌、涨停买不到、跌停卖不掉、集合竞价撤单导致的虚假强度，仍没有完整建模。

### P0: 策略闭环

- 普通模型已有入场计划，但日常流程尚未写入 `strategy_signal_log`。
- 人工是否采纳、跳过原因、实际成交价、卖出原因，没有自动形成 `strategy_trade_journal`。
- 没有每天的“计划 vs 实际 vs 偏差”复盘表。

### P0: 验证方式

- 现有很多搜索结果仍带样本内优化味道。
- Relay 参数扫描明确写着是样本内方向，不是上线结论。
- 上线门槛还没被程序化执行：样本外交易数、最大回撤、连续亏损、月亏损、单笔亏损、收益集中度。

### P1: 工程可复现

- 当前裸环境缺 `numpy/pandas/sqlalchemy/pymysql/baostock/akshare` 等依赖，不能一键复跑回测。
- `config.ini` 中有本地 MySQL 明文连接信息。
- `tgBot.py` 有硬编码 Telegram token fallback，必须移到环境变量并轮换 token。
- `.gitignore` 太弱，`reports/` 已经 433MB，生成物、数据库、PDF、pyc 都没有被系统性隔离。

## 4. 模型分级

| 模型 | 实盘定位 | 当前决定 |
| --- | --- | --- |
| LAOWANG | 低频观察池 | 暂不实盘，只保留扫描 |
| STWG | 平台突破观察池 | 只允许突破确认后小仓纸面验证 |

| FHKQ | 极端事件监控 | 不做常规交易 |
| Relay | 次日接力候选 | 主攻，但必须接入竞价/分钟线 |

## 5. 下一步抢救路线

### 第 1 周：先把手动实盘闭环补上

- 生成 `next_day_plan` 和 `relay_watchlist` 后，同步写入 `strategy_signal_log`。
- 增加一个人工回填脚本或 UI 表单：成交/未成交、成交价、仓位、跳过原因、卖出价、卖出原因。
- 每天输出一张 `daily_execution_audit`：计划、实际、偏差、风险暴露。
- 去掉 Telegram 硬编码 token，清理明文数据库配置，补 `.env.example`。

### 第 2 周：把 Relay 变成真正的盯盘系统

- 创建分钟/竞价衍生特征表：`auction_gap`、`auction_amount_ratio`、`first_5m_ret`、`first_15m_ret`、`vwap_reclaim`、`re_seal_flag`、`intraday_drawdown`。
- Relay 输出改成两段：盘后候选和 T+1 盘中人工触发表。
- 无分钟线时明确标记 `data_mode=daily_proxy`，不允许把代理结果当实盘证据。

### 第 3 到 4 周：样本外验收

- 固化一套 Relay 参数，不再每天搜索最优。
- 跑 walk-forward：训练、验证、测试严格按时间切开。
- 输出分月、分市场阶段、分连板高度、分竞价 gap 的收益和回撤。
- 只有达到门槛才允许小资金：
  - 样本外交易数不少于 100。
  - 双倍成本后收益仍为正。
  - 最大回撤不超过 12% 到 15%。
  - 单笔最大亏损不超过 5%。
  - 连续亏损不超过 5 笔。
  - 任一单笔盈利不能贡献超过总收益 40%。

## 6. 结论

这个 repo 不是纯废物，它已经有日常数据、评分、UI、推送、回测和初步整改方案。真正的问题是它目前仍是“选股雷达 + 研究报告生成器”，不是“交易系统”。

最短可走路径：

1. 普通模型不实盘，最多观察。
2. Relay 做主攻，但先只做人工盯盘候选。
3. 先跑 30 到 60 个交易日纸面/极小仓，强制记录每次人工决策。
4. 没有真实分钟线和交易日志前，不加仓，不自动下单。

如果目标是隔日手动下单，项目离“可开始小资金受控验证”不算远；如果目标是稳定实盘赚钱，它还差一整套执行、审计、风控和样本外验证。
